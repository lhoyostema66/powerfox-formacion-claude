Ecosistema Power Platform — Los 5 pilares

S1-Lunes · Bloque 1

Los 5 pilares

  • Power Apps — Crea interfaces de usuario. Dos tipos: Model-Driven (sigue la estructura de Dataverse) y Canvas (diseño libre).
  • Power Automate — Automatiza flujos y procesos. Se conecta con más de 1.000 servicios externos.
  • Power BI — Análisis y visualización conectada directamente a los datos de Dataverse.
  • Copilot Studio — Agentes de IA con acceso a datos empresariales.
  • Power Pages — Portales web para usuarios externos, también sobre Dataverse.

La clave que une todo: Dataverse

Todos los pilares convergen en Microsoft Dataverse, que es la capa de datos. Sobre él se construye todo el sistema: tablas, relaciones, permisos y lógica de negocio.

Una imagen para recordarlo: Dataverse es como el suelo de un edificio. Power Apps, Power Automate y Power BI son las habitaciones que se construyen encima. Sin suelo, no hay edificio.

Modelo de datos y Dataverse

S1-Lunes · Bloques 2 y 3 · S1-Miércoles · Bloque 3

Tipos de tablas

  • Standard — Predefinidas por Microsoft (Account, Contact...).
  • Custom — Las que crea el equipo para el proyecto (prefijo pfx_).
  • Virtual — Conectan datos externos sin migrarlos.
  • Elastic — Para grandes volúmenes de datos.

Tipos de relaciones

  • 1:N (uno a muchos) — La más común. Ej: un Proyecto tiene muchas Actividades. La entidad "hija" tiene un campo Lookup al "padre".
  • N:N (muchos a muchos) — Tabla intermedia automática. Ej: un Contacto puede tener varias Habilidades.
  • 1:1 (uno a uno) — Se simula con un Lookup único.

Entornos y ALM — DEV / TEST / PROD

S1-Miércoles · Bloque 1

¿Por qué múltiples entornos?

  • DEV — Aquí se construye y prueba sin riesgo. Los errores no afectan a nadie.
  • TEST — Se valida con datos reales de prueba. Los usuarios pueden probar sin miedo.
  • PROD — El entorno real. Solo llega aquí lo que ha pasado DEV y TEST.

Para ti como técnica: cuando un cambio "no aparece todavía", es porque está en DEV o en TEST esperando validación. No es un bug, es el proceso.

Managed vs Unmanaged

TipoDónde¿Editable?
UnmanagedSolo en DEVSí, libre
ManagedTEST, UAT, PRODNo — bloqueada

En PROD siempre soluciones managed. Así se puede quitar todo de golpe si algo falla.

El viaje de un cambio — de la idea a producción

1. Petición → 2. Diseño → 3. DEV → 4. TEST → 5. UAT → 6. PROD

Formularios Model-Driven App

S1-Miércoles · Bloque 4 · S1-Viernes · Bloques 1, 2 y 3

Tipos de formularios

  • Main Form — El formulario principal de edición de un registro.
  • Quick View Form — Muestra datos del registro padre dentro del hijo sin necesidad de código.
  • Card Form — Vista compacta para listas de tarjetas.

Truco útil: Formula selector para mostrar formularios distintos según el rol del usuario. Un gerente puede ver campos económicos que un técnico no ve.

Business Rules — lógica sin código

Antes de llegar a JavaScript, las Business Rules permiten:

  • Mostrar u ocultar campos según condiciones
  • Hacer un campo obligatorio si otro tiene un valor determinado
  • Establecer valores por defecto

Son visuales, sin código, y suficientes para mucha lógica sencilla. Si Business Rule puede hacerlo, úsala primero.

Subgrids

Muestran registros relacionados dentro de un formulario. Desde un Proyecto puedes ver directamente sus Actividades o sus Partes de Horas sin salir del formulario.

Vistas

  • System Views — Compartidas con todos los usuarios.
  • Lookup Views — Controla qué columnas ve el usuario al buscar en un lookup.
  • Quick Find View — Define qué campos son buscables.

JavaScript en formularios

S1-Miércoles · Bloque 6 · S1-Viernes · Bloques 1, 2, 3 y 4

Este fue el bloque que te costó más. El objetivo aquí es que te quede el esqueleto claro, no que lo domines aún. JavaScript en formularios se consolida practicando.

¿Cuándo se usa JavaScript?

Solo cuando Business Rule no llega:

  • Validaciones complejas con varias condiciones
  • Consultar datos de otra tabla al abrir el formulario
  • Filtrar lookups dinámicamente según otro campo
  • Lógica que necesita cálculos en tiempo real

Limitación importante: JavaScript solo se ejecuta en el formulario del navegador. Si alguien crea un registro desde la API o un flow, el JS no se ejecuta. Para lógica que debe aplicarse siempre, necesitas Plugin o Power Automate.

Los 3 eventos básicos

EventoCuándo
OnLoadAl abrir el formulario — inicializar, ocultar campos
OnChangeAl modificar un campo — calcular, auto-rellenar
OnSaveAntes de guardar — validar, prevenir guardado si hay error

La puerta de entrada: formContext

formContext es el "mando a distancia" del formulario. Con él puedes leer y escribir campos, ocultar secciones, y notificar al usuario. Siempre se obtiene del executionContext — nunca de Xrm.Page (que está obsoleto).

// Patrón base — siempre así function onLoad(executionContext) { const fc = executionContext.getFormContext(); // Leer un valor const nombre = fc.getAttribute("pfx_name").getValue(); // Ocultar un campo fc.getControl("pfx_campo").setVisible(false); // Hacer obligatorio fc.getAttribute("pfx_email").setRequiredLevel("required"); // Mostrar notificación al usuario fc.ui.setFormNotification("Falta información", "WARNING", "id_aviso"); }

Xrm.WebApi — cuando necesitas datos de otra tabla

Para leer datos de otras tablas desde el formulario se usa Xrm.WebApi. Hoy conociste los dos métodos básicos:

// Leer un registro concreto Xrm.WebApi.retrieveRecord( "pfx_project", "guid-del-registro", "?$select=pfx_project_code,pfx_project_name" ).then(function(result) { console.log(result.pfx_project_name); });

Skills y Agents en Claude Code

S1-Viernes · Bloque 6 (bloque estrella)

¿Qué es una Skill?

Instrucciones que Claude Code aplica automáticamente cada vez que genera código nuevo. No tienes que recordarle las reglas en cada prompt.

La skill MDA JavaScript Writer creada en sesión garantiza que el código nuevo nazca con buenas prácticas: const/let en lugar de var, null checks, namespace pattern, JSDoc.

¿Qué es un Agent?

Un especialista invocable. A diferencia de la skill (siempre activa), el agent se llama cuando lo necesitas:

  • MDA Code Reviewer — Analiza código existente y genera un informe de issues sin modificar nada.
  • MDA Refactor Assistant — Toma el código existente y lo reescribe aplicando las mejoras, explicando cada cambio.

Por qué importa para ti: Con 0 experiencia en JS, los agents son tu red de seguridad. Cuando Claude Code te genere un script, puedes pasarlo al Code Reviewer para verificar que está bien antes de publicarlo. No necesitas saber revisarlo tú misma todavía.

Puntos clave para tu rol

Filtrado desde la perspectiva de Técnica de proyectos y Laboratorio. Lo que más te puede servir en tu día a día, conectado con trazabilidad de ensayos, seguimiento de proyectos y gestión de laboratorio.

Ecosistema: lo que necesitas recordar

  • Power Apps Model-Driven + Dataverse es la combinación para gestionar proyectos y registros de laboratorio con trazabilidad completa y permisos controlados.
  • Power BI se conecta directamente: si los datos de ensayos están en Dataverse, los puedes visualizar y analizar sin exportaciones manuales a Excel.
  • Power Automate puede notificar automáticamente cuando un ensayo cambia de estado o cuando vence un plazo — sin que nadie tenga que acordarse de enviar el email.

Modelo de datos: la base del sistema

  • Una tabla en Dataverse es como una hoja de Excel con lógica: columnas tipadas, relaciones a otras tablas, y reglas de negocio encima.
  • Para registros de ensayos, los campos Formula 🔒 son útiles para calcular métricas automáticas (ej: margen, porcentaje de avance) sin que nadie los introduzca manualmente.
  • El prefijo pfx_ en todos los campos del proyecto es una buena práctica de identificación que puedes aplicar si creas tus propias tablas de laboratorio.

Entornos ALM: lo que cambia en tu contexto

  • Como técnica, cuando alguien dice "está en TEST", significa que el cambio existe pero aún no llegó a producción. Normal, no un error.
  • Si algún día propones una mejora en el sistema (un campo nuevo, una vista distinta), pasará por DEV → TEST → PROD antes de ser visible. El proceso tarda pero protege los datos reales.

Herramientas: por dónde empezar

  • FetchXML Builder es la herramienta de análisis más accesible para ti ahora mismo. Puedes hacer consultas tipo "proyectos activos sin actividades" o "ensayos del mes pasado por tipo" sin código.
  • Claude Code como chat avanzado: ya lo usas para escritura. El siguiente paso es usarlo para interpretar código que no entiendes, o para pedir que te explique un concepto de la plataforma con ejemplos de tu área.

JavaScript: solo el esqueleto por ahora

  • No necesitas escribir JS hoy. Lo que sí debes tener claro: JS solo vive en el formulario del navegador — si alguien importa datos, el JS no se ejecuta.
  • El prompt que guardas en tu bitácora ("explícame qué hace este código y si hay algo que mejorar") es exactamente el uso correcto para tu nivel actual.
  • Cuando llegues a necesitar una validación en un formulario, Business Rules llega lejos sin código.

Skills y Agents: tu ventaja desde el primer día

  • Con 0 experiencia en JS, los agents son tu red de seguridad para el código que generes o que recibas de Claude Code.
  • La skill MDA JavaScript Writer asegura que el código que Claude te genere ya sea correcto desde el principio.
  • Para un uso de escritura y revisión (que ya haces), Skills con instrucciones de estilo o terminología técnica de laboratorio podrían serte útiles en las próximas sesiones.

Ejercicio 2 · Consultas

Consulta de proyectos sin avance reciente

Con FetchXML Builder, crea una consulta que devuelva los proyectos activos que no tienen ningún registro de avance en el último mes.

Esto simula un caso real: ¿qué proyectos están "olvidados"?

  • Tabla: pfx_project
  • Filtro: estado activo
  • Left join con pfx_project_progress
  • Condición: sin registros en los últimos 30 días

Si te atascas, describe el objetivo a Claude y pide que te genere el FetchXML.

Ejercicio 3 · Formularios

Business Rule para un formulario de laboratorio

Imagina que tienes un formulario de registro de ensayo con campos "Fecha inicio" y "Fecha fin". Crea una Business Rule (sin código) que:

  • Haga obligatorio "Fecha fin" cuando el estado sea "Completado"
  • Muestre un aviso si "Fecha fin" es anterior a "Fecha inicio"

Puedes hacerlo en el entorno DEV creando una tabla de prueba, o pedirle a Claude que te explique exactamente los pasos paso a paso.

Notas sobre tu evolución

Foix arranca el curso desde cero en Power Platform pero con una base real que no siempre se tiene: curiosidad activa, hábito de indagar, y ya un uso cotidiano de Claude para escritura y revisión. Ese punto de partida es más útil de lo que parece.

En la primera semana el ecosistema de herramientas quedó claro — lo destacas explícitamente como "lo mejor" de S1 y lo confirmaste en sesión. Saber para qué sirve cada pieza (Power Apps, Dataverse, Power Automate, FetchXML Builder) es la base sobre la que todo lo demás se apoya.

Los dos bloques que costaron más — la arquitectura general y JavaScript — tienen en común que son conceptos que no se asimilan leyendo, sino tocando. La arquitectura por capas (presentación / lógica / datos / integración / seguridad) solo se consolida cuando ves cómo un cambio en una capa afecta a las demás. JavaScript, cuando escribes tu primer script y lo ves ejecutarse en el formulario.

El prompt que guardaste en la bitácora ("explícame qué hace este código") es exactamente el uso de IA correcto para tu nivel y tu objetivo: no generar código ciegamente, sino entender el que ya existe. Eso es más maduro de lo que muchas personas con más experiencia hacen.

El reto para las próximas sesiones es pasar de observadora a ejecutora: el primer ensayo en DEV, la primera consulta en FetchXML Builder, el primer Business Rule. Con 0 experiencia previa, ese primer contacto práctico es el salto que marca la diferencia.
