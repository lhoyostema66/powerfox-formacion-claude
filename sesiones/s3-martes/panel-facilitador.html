01

Qué quiero que se lleven

3 cosas, no más

  1. Que sepan crear un proyecto React desde cero pidiéndoselo a Claude. No memorizando comandos npm, sino sabiendo cómo encargar un proyecto.
  2. Que entiendan el ciclo "prompt → revisar → iterar". Claude no acierta a la primera. Lo importante es saber leer su salida y corregir.
  3. Que cada uno tenga una URL pública con su Weather Report al final. Es el "trofeo" que se llevan a casa.

Qué NO quiero hacer hoy

  • No entrar en CSS pixel-perfect. Si Claude lo deja "feo pero funcional", suficiente para hoy.
  • No explicar React en profundidad. La app funciona. Si quieren entender hooks, eso lo ven en otro contexto.
  • No meter TypeScript. JavaScript plano. Reduce fricción para los no técnicos.
Trampa habitual: querer "explicar el código" cuando Claude lo genera. Si el alumno no lo pide, no lo explicas. La curiosidad se la guardan ellos.

P1 — Crear la app entera con un prompt

0:30 → 1:00 · 30 min

Este es el momento clave del martes. Pegan el prompt, ven cómo Claude crea archivos durante varios minutos, y al final tienen una app que funciona.

0:30P1 — Scaffolding completo⭐ ESTRELLA

Qué hace este prompt

  • Crea un proyecto Vite + React desde cero.
  • Genera los componentes SearchBar, WeatherCard, ForecastList.
  • Usa datos mockeados al principio (no toca la API real todavía).
  • CSS plano (no Tailwind).
  • Deja el proyecto listo para npm install + npm run dev.
P1 Crea la app Weather Report
Crea una aplicación React + Vite llamada "weather-report" en la carpeta actual. Requisitos: ESTRUCTURA: - Usa Vite con plantilla react (NO TypeScript, JavaScript plano). - src/App.jsx como componente raíz. - src/components/SearchBar.jsx — input para escribir una ciudad y un botón "Buscar". - src/components/WeatherCard.jsx — tarjeta que muestra: ciudad, temperatura actual, descripción, humedad, viento. - src/components/ForecastList.jsx — lista vertical de 7 días con icono, día de la semana, máxima y mínima. - src/api/weather.js — funciones para obtener datos. POR AHORA devuelve datos mockeados (no llames a ninguna API real todavía). DATOS MOCK: - ciudad: "Madrid" - temperatura: 22°C - descripción: "Soleado" - humedad: 45%, viento: 12 km/h - forecast de 7 días con valores de ejemplo ESTILOS: - CSS plano en App.css y un módulo por componente si hace falta. - NO uses Tailwind, ni styled-components, ni librerías de UI. - Diseño limpio y centrado, fondo claro, bordes suaves, paleta azul/blanco. - Responsive: en móvil (<600px) que se vea bien apilado. INSTALACIÓN: - Crea el package.json con scripts dev, build, preview. - NO ejecutes npm install todavía — deja el proyecto listo para que yo lo lance. DOCUMENTACIÓN: - Genera un README.md corto explicando cómo arrancar (npm install, npm run dev) y qué hace cada componente. REGLAS DURAS: - No inventes APIs todavía — datos mock siempre. - Comentarios mínimos, solo donde el "por qué" no sea obvio. - Sin console.log innecesarios. - En español el README, en inglés solo lo que tenga que ser estándar (nombres de funciones).

Qué pasa después de pegar P1

  1. Claude empieza a crear archivos. Verás varias acciones de "Write" en el panel.
  2. Al terminar, te muestra el árbol de archivos creados.
  3. Tú ejecutas en terminal:
    • npm install (tarda ~30 seg)
    • npm run dev
  4. Vite arranca en http://localhost:5173. Lo abres en el navegador.
  5. Si funciona — aplauso colectivo y siguiente bloque. Si no — saltas a Troubleshooting.
Cómo enseñarlo

Mientras Claude trabaja (tarda 1-2 minutos), no te quedes en silencio. Explica qué está pasando: "Claude está creando los archivos uno a uno. Es como si le hubieras pedido a un becario que monte el proyecto, pero el becario tarda 90 segundos en vez de 4 horas."

Cuando salga el árbol de archivos, abre cada uno en VS Code y muéstralo 10 segundos. Sin leer el código en voz alta. Solo "esto es lo que pinta la tarjeta", "esto es la barra de búsqueda".

Si Claude propone Tailwind o TypeScript: respondes "no, JavaScript plano y CSS plano", y reintenta. No discutas el por qué con la clase — luego en privado les explicas que Tailwind hubiera complicado el setup.
B4

P2 — Iterar UX con Claude

1:40 → 2:20 · 40 min

La app ya funciona pero está incompleta. Aquí enseñamos lo más importante: iterar. No tirar y rehacer — pedir cambios concretos.

1:40P2 — Mejorar barra de búsqueda + estados⭐ ITERACIÓN
P2 Iterar UX: búsqueda real + estados de error
La app ya funciona con datos mock. Ahora mejórala: 1. La barra de búsqueda debe permitir escribir una ciudad y al pulsar "Buscar" o presionar Enter, actualizar la WeatherCard. Por ahora sigue usando datos mockeados, pero crea un mock distinto por cada ciudad: "Madrid", "Barcelona", "Lima", "Sevilla". Si el usuario escribe otra ciudad, muestra un estado de "no encontrada". 2. Añade tres estados visuales: - "Cargando..." mientras se "busca" (simula 500ms con setTimeout). - "Ciudad no encontrada" si no está en el mock. - Estado normal con la tarjeta del clima. 3. Si el input está vacío y pulsa Buscar, muestra un mensaje "Escribe el nombre de una ciudad". 4. Pequeñas mejoras de UX: - Foco automático en el input al cargar la página. - Botón "Buscar" deshabilitado si el input está vacío. - Limpiar el input tras una búsqueda exitosa. 5. NO toques aún la API real. Esto sigue siendo mock. Mantén el estilo CSS plano. No introduzcas librerías nuevas.

Qué enseñar mientras Claude trabaja

  • El concepto de "estados" en UI: cargando, vacío, con datos, con error. Sin esto, una app parece rota cuando no es rota.
  • Cómo Claude hace cambios incrementales: no recrea el proyecto, edita los archivos existentes. Lo ven en el panel.
  • Hot reload de Vite: al guardar Claude los archivos, la app se actualiza sola en el navegador. "Mirad, no he tocado nada y ya cambió."
Tip didáctico: después de aplicarse el cambio, escribes en el input "Cuenca" (que NO está en el mock) y enseñas el estado "no encontrada". Luego "Madrid" y enseñas el cargando + datos. Es teatro útil — ven los 3 estados sin que tengan que tocar nada.

P3 + P4 — Conectar a API real (miércoles)

Miércoles 0:15 → 1:45 · ~90 min

Este es el bloque "wow" del miércoles. Pasamos de datos inventados a datos reales del clima del momento. Y lo hacemos sin escribir una línea de código.

0:15P3 — Sustituir mock por Open-Meteo⭐ API REAL
P3 Conectar a Open-Meteo (sin API key)
Sustituye los datos mockeados de src/api/weather.js por llamadas reales a Open-Meteo. Open-Meteo es una API gratuita y SIN API KEY. Documentación: https://open-meteo.com/en/docs Necesitas dos llamadas: 1. GEOCODING — para convertir el nombre de la ciudad a coordenadas: GET https://geocoding-api.open-meteo.com/v1/search?name=Madrid&count=1&language=es&format=json Respuesta relevante: results[0].latitude, results[0].longitude, results[0].country, results[0].name. 2. CLIMA ACTUAL — con esas coordenadas: GET https://api.open-meteo.com/v1/forecast?latitude=40.4&longitude=-3.7¤t_weather=true&hourly=relative_humidity_2m&timezone=auto De ahí sacas: current_weather.temperature, current_weather.windspeed, current_weather.weathercode, hourly.relative_humidity_2m[0]. Implementa una función getWeatherByCity(city) que: - Llame a geocoding. - Si no devuelve resultados, lance un error con mensaje "Ciudad no encontrada". - Si lo devuelve, llame a forecast con las coordenadas. - Devuelva un objeto normalizado: { city, country, temperature, description, humidity, windspeed }. Mapea el weathercode a una descripción en español: - 0: "Despejado" - 1, 2, 3: "Parcialmente nublado" - 45, 48: "Niebla" - 51, 53, 55: "Llovizna" - 61, 63, 65: "Lluvia" - 71, 73, 75: "Nieve" - 95: "Tormenta" - otros: "Variable" Actualiza App.jsx para que use esta función en vez del mock. Mantén el manejo de estados (cargando, error, normal) que ya tenemos. NO uses axios ni librerías HTTP — usa fetch nativo. NO añadas claves de API ni variables de entorno — esta API no las necesita.

Probar después de P3

  1. Vite recarga automáticamente.
  2. Escribes "Lima" → datos reales del clima de Lima ahora mismo.
  3. Escribes "asdfg" → estado "Ciudad no encontrada".
  4. Mensaje a la clase: "Ese 22°C no me lo he inventado. Es lo que está pasando ahora mismo en la calle."
1:05P4 — Pronóstico semanal + pulir UI⭐ PULIR
P4 Forecast 7 días + UX final
Mejora la app con: 1. PRONÓSTICO 7 DÍAS — añade a la llamada de Open-Meteo el parámetro daily: &daily=temperature_2m_max,temperature_2m_min,weathercode Llena la ForecastList con esos 7 días. Cada fila muestra: - Día de la semana en español ("Lun", "Mar", etc.) - Icono o emoji según el weathercode (☀️ ☁️ 🌧️ ❄️ ⛈️) - Temperatura máx / mín 2. UX FINAL: - Al cargar la página, busca automáticamente "Madrid" y muestra esos datos (estado por defecto, no input vacío). - Persiste la última ciudad buscada en localStorage para que al recargar muestre la misma. - Añade un botón pequeño "📍 Mi ubicación" que use navigator.geolocation para coger lat/lon del navegador y consultar Open-Meteo directo (sin geocoding). 3. ESTILO: - Asegúrate de que en móvil (max-width: 600px) la ForecastList se vea apilada y las fuentes sean legibles. - Pequeñas animaciones suaves al cambiar de ciudad (fade in 200ms). 4. README.md — actualízalo con: - Cómo correr la app. - Qué API usa y por qué (sin key). - Decisiones de diseño tomadas. Mantén CSS plano. No añadas nuevas dependencias.

Probar después de P4

  • App al cargar muestra Madrid con su forecast 7 días.
  • Click en "📍 Mi ubicación" → permiso del navegador → muestra clima local.
  • Recargar la pestaña recuerda la última búsqueda.

Vercel — setup colectivo

Lo más rápido es que tú conectes el repo del curso a tu cuenta Vercel personal y configures un deploy por subcarpeta. Vercel detecta cualquier push a main y despliega demos/weather-report.

Pero como cada alumno tiene su versión en su rama propia, lo elegante es:

  • Cada alumno crea su propio fork del repo a su cuenta GitHub personal (si no lo tiene ya).
  • Conecta su fork a Vercel (gratis, login con GitHub, "Import Project").
  • Vercel le da una URL tipo powerfox-formacion-claude-LUIS.vercel.app.
P5 Configurar Vercel para deploy de la app
Voy a desplegar la app weather-report a Vercel. La app vive en demos/weather-report dentro de un repo monorepo. Necesito: 1. Crear un archivo vercel.json en la raíz del repo que indique a Vercel que el "root directory" del proyecto es demos/weather-report. Que use Vite (build command: npm run build, output directory: dist). 2. Asegurarme de que la app se construye correctamente con npm run build: - Verifica que vite.config.js no tenga base path raros. - Verifica que no haya rutas absolutas hardcoded. 3. Añadir una sección "Deploy" al README.md de demos/weather-report con instrucciones cortas: - Login en vercel.com con GitHub. - Click en "Import Project". - Pegar la URL del fork del alumno. - Configurar Root Directory: demos/weather-report. - Deploy. 4. NO subas a Vercel desde aquí — solo prepara la configuración. El alumno hace el deploy desde la web de Vercel.
